- **Linting:** ESLint + Prettier

## Centralized Configuration

All service configuration is managed via `config.toml` at the project root.

### Configuration Flow

```
┌─────────────────────────────────────────────────────────────────────┐
│                        config.toml                                   │
│                    (Source of Truth)                                 │
└───────────────────────────┬─────────────────────────────────────────┘
                            │
        ┌───────────────────┼───────────────────┐
        │                   │                   │
        ▼                   ▼                   ▼
┌───────────────┐   ┌───────────────┐   ┌───────────────────┐
│ Local Dev     │   │ Containers    │   │ Cloud Run         │
│               │   │               │   │                   │
│ Config loader │   │ .env.generated│   │ Env vars set via  │
│ reads TOML    │   │ + compose.yaml│   │ gcloud/terraform  │
│ directly      │   │               │   │                   │
└───────────────┘   └───────────────┘   └───────────────────┘
```

### Config Resolution Order (later wins)

1. `[default.*]` sections in config.toml
2. `[production.*]` sections (environment-specific)
3. Environment variables (`SOMNIA_AGENT_HOST`, `SOMNIA_AGENT_PORT`, `SOMNIA_AGENT_URL`, etc.)

### Key Files

| File | Purpose |
|------|---------|
| `config.toml` | Master configuration - services, agents, networks, LLM |
| `scripts/generate-env.ts` | Generates `.env.generated` from config.toml |
| `.env.generated` | Auto-generated env file for compose.yaml |
| `packages/shared/src/config/` | TypeScript config loader module |
| `packages/agent-registry/` | Service that reads config and serves agent list |

## Somnia Chain Info

Somnia is a high-performance EVM-compatible L1:
- **Performance:** 1,000,000+ TPS, sub-second finality
- **Consensus:** MultiStream Consensus
- **Mainnet:** Chain ID 5031, RPC: https://api.infra.mainnet.somnia.network/
- **Testnet (Shannon):** Chain ID 50312, RPC: https://dream-rpc.somnia.network/
- **Native Token:** SOMI (mainnet), STT (testnet)
- **Unique Features:** Reactivity (on-chain pub/sub), Data Streams

### Gas Considerations
- LOG operations: Higher cost (8,320+ gas vs Ethereum's 631)
- Contract deploy: 3,125 gas/byte (vs 200)
- ecRecover: 150,000 gas (vs 3,000)
- Cold storage: +1M gas penalty outside 128M key cache

## Known Issues & Investigations

### Issue: A2A Tools Load in TUI but Not Web Frontend (2026-02-11)

**Status:** Fix Implemented - Separate Server Ports

**Symptom:** The A2A plugin tools (`a2a-send`, `a2a-agents`, `a2a-capabilities`) work when using `opencode` TUI but are not available when using the web frontend (`opencode web`).

**Resolution:** The issue was caused by running two OpenCode instances on the same port (4096). The current OpenCode TUI session and the opencode-server for A2A testing were conflicting. 

**Fix:** Configure opencode-server to run on port 4097 (via Makefile) and frontend to connect to 4097 (via config.toml). See "Configuration" section above.

**Root Cause Analysis:**

Both TUI and Web use the same backend server (`packages/opencode/src/server/server.ts`). The key components:

1. **Plugin Registration Path:**
   ```
   server.ts middleware (line 220-237)
     → Instance.provide({ init: InstanceBootstrap })
       → InstanceBootstrap() (bootstrap.ts:17)
         → Plugin.init() (plugin/index.ts:136)
           → loads INTERNAL_PLUGINS including A2APlugin
   ```

2. **Tool Loading Path:**
   ```
   SessionPrompt (prompt.ts:720)
     → ToolRegistry.tools()
       → ToolRegistry.state() (registry.ts:34)
         → Plugin.list() (plugin/index.ts:132)
           → returns hooks from all loaded plugins
   ```

3. **Key Difference Between TUI and Web:**

   | Aspect | TUI (worker.ts) | Web (web.ts) |
   |--------|-----------------|--------------|
   | Bootstrap | Immediate via `startEventStream(process.cwd())` at line 97 | Lazy on first HTTP request |
   | Directory | Uses `process.cwd()` | Uses `x-opencode-directory` header from frontend |
   | Instance State | Cached per directory | Cached per directory |

4. **Potential Issues:**
   - **Directory Mismatch:** Web frontend sends directory via `x-opencode-directory` header (SDK `client.ts:21-28`). If this directory differs from TUI's `process.cwd()`, separate Instance states are created, each loading plugins independently.
   - **Race Condition:** `ToolRegistry.state()` calls `Plugin.list()`. If accessed before `Plugin.init()` completes, plugins may not be registered.
   - **Plugin Error:** If A2A plugin throws during initialization (caught at `plugin/index.ts:54-59`), it won't be registered but error is only logged.

**Investigation Steps:**
1. Check server logs for "loading internal plugin" + "a2a" when web frontend makes requests
2. Verify directory being sent from web frontend matches expected value
3. Call `/experimental/tool/ids` endpoint from both TUI and web to compare registered tools
4. Check for any errors during plugin initialization

**Relevant Files:**
- `opencode/packages/opencode/src/server/server.ts` - Server middleware (line 220-237)
- `opencode/packages/opencode/src/project/bootstrap.ts` - Instance bootstrap
- `opencode/packages/opencode/src/plugin/index.ts` - Plugin loading
- `opencode/packages/opencode/src/tool/registry.ts` - Tool registry
- `opencode/packages/opencode/src/cli/cmd/web.ts` - Web command
- `opencode/packages/opencode/src/cli/cmd/tui/worker.ts` - TUI worker

## Unified LLM Configuration

All LLM configuration is centralized in `config.toml`. Each component can reference its preferred LLM provider, with a default fallback.

### Config Structure

```toml
# config.toml

# Default provider for components not explicitly assigned
[default.llm]
default = "vertex-gemini"

# Available providers
[default.llm.providers.vertex-gemini]
type = "vertex"
project = "kunal-scratch"
location = "us-central1"
model = "gemini-2.0-flash"

[default.llm.providers.vertex-anthropic]
type = "vertex-anthropic"
project = "kunal-scratch"
location = "us-east5"
model = "claude-sonnet-4"

# Component assignments (omitted = uses default)
[default.llm.components]
somnia-agent = "vertex-gemini"
sonic-agent = "vertex-gemini"
midnight-agent = "vertex-gemini"
opencode = "vertex-anthropic"
```

### How It Works

1. **`config.toml`** defines reusable LLM providers and assigns them to components
2. **`scripts/generate-env.ts`** reads config.toml and generates `.env.generated`:
   - Component-specific vars: `SOMNIA_AGENT_LLM_PROJECT`, `SOMNIA_AGENT_LLM_LOCATION`, etc.
   - OpenCode-specific vars: `GOOGLE_CLOUD_PROJECT`, `VERTEX_LOCATION`, `GOOGLE_CLOUD_LOCATION`
   - **Important:** `GOOGLE_CLOUD_LOCATION` is set to OpenCode's provider location (not default)
     because OpenCode reads `GOOGLE_CLOUD_LOCATION` before `VERTEX_LOCATION`
3. **`compose.yaml`** loads `.env.generated` via `env_file`
4. **Components** read from their component-specific env vars, with fallback to generic vars

### Generated Environment Variables

```bash
# .env.generated (auto-generated)

# somnia-agent → vertex-anthropic
SOMNIA_AGENT_LLM_TYPE=vertex-anthropic
SOMNIA_AGENT_LLM_PROJECT=kunal-scratch
SOMNIA_AGENT_LLM_LOCATION=global
SOMNIA_AGENT_LLM_MODEL=claude-opus-4.5

# opencode → vertex-anthropic
OPENCODE_LLM_TYPE=vertex-anthropic
OPENCODE_LLM_PROJECT=kunal-scratch
OPENCODE_LLM_LOCATION=global
OPENCODE_LLM_MODEL=claude-opus-4.5

# OpenCode-specific (reads these env vars)
# GOOGLE_CLOUD_LOCATION is set to OpenCode's provider location
# because OpenCode reads GOOGLE_CLOUD_LOCATION before VERTEX_LOCATION
GOOGLE_CLOUD_PROJECT=kunal-scratch
VERTEX_LOCATION=global
GOOGLE_CLOUD_LOCATION=global
```

### Why Different Regions?

| Provider | Model | Supported Regions |
|----------|-------|-------------------|
| `vertex` | Gemini | `us-central1`, `us-east4`, etc. |
| `vertex-anthropic` | Claude | `us-east5`, `europe-west1`, etc. |

Claude via Vertex AI is only available in specific regions (`us-east5`, `europe-west1`), while Gemini works in most regions.

### Adding a New Component

1. Add to `config.toml`:
   ```toml
   [default.llm.components]
   my-new-agent = "vertex-gemini"  # or create a new provider
   ```

2. Run `bun scripts/generate-env.ts` to regenerate `.env.generated`

3. In your component code, read from env vars:
   ```typescript
   const project = process.env.MY_NEW_AGENT_LLM_PROJECT || process.env.GOOGLE_CLOUD_PROJECT;

   ```

### IAP Authentication (Google Sign-In)

OpenCode web uses Google's Identity-Aware Proxy (IAP) to restrict access to whitelisted users.

#### How It Works

```
User visits opencode-web URL
    ↓
IAP: Is user authenticated? → NO → Redirect to Google Sign-In
    ↓
IAP: Is user in whitelist? → NO → "Access denied" page
    ↓
YES → OpenCode loads, /global/me returns user email
```

#### User Management

Allowed users are defined in `config.toml`:

```toml
[default.auth]
allowed_users = [
  "user1@gmail.com",
  "user2@example.com",
]
```

**To add/remove users:**
```bash
# 1. Edit config.toml
# 2. Sync to IAP
make cloud-sync-users
```

#### Local Development

When running locally (no IAP), the `/global/me` endpoint uses `gcloud config get account` to get the developer's Google account. This provides a consistent experience between local and production.

#### Files

| File | Purpose |
|------|---------|
| `config.toml` → `[default.auth]` | Allowed users list |
| `scripts/cloud.sh sync-users` | Syncs config.toml to IAP bindings |

| `opencode/.../context/auth.tsx` | Frontend auth context |
